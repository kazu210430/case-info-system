# AGENTS.md

## 1. 役割

このファイルは、AI コーディングエージェントが案件情報Systemを修正する際の行動ルールを定義するためのものです。

このルールは、現行 `main` を基準点とし、`docs/architecture.md`、`docs/flows.md`、`docs/ui-policy.md` を前提として扱います。

## 2. 作業前ルール

作業前に必ず以下を読むこと。

- `docs/architecture.md`
- `docs/flows.md`
- `docs/ui-policy.md`

作業前に必ず以下を確認すること。

- 対象機能のフローを理解してから修正する
- 関連サービスを特定してから変更する
- 記述と判断は事実ベースで行う
- 不明点は補完せず、確認が必要な事項として扱う

修正に入る前に、必ず以下の思考宣言を出力すること。

- 対象フローの要約
- 修正対象サービスの特定
- 修正範囲（どのファイル・どの処理を変更するか）

この出力なしにコード修正へ進むことを禁止する。

## 3. 修正ルール

- `docs/architecture.md` にある構成と責務を壊さない
- `docs/flows.md` にある対象機能のフローを確認してから修正する
- `docs/ui-policy.md` にある UI 制御方針に従う
- build は必ずリポジトリ直下の `build.ps1` 経由で実行する
- build / test / Debug Add-in 配備の確認は、原則としてリポジトリ直下の `build.ps1` を標準入口として使う
- `dotnet build .\dev\CaseInfoSystem.slnx` を標準確認コマンドとして使用しない。VSTO packaging ガードは安全装置として維持する
- Compile = 「CI相当の安全なビルド確認」として扱う
- Compile では Add-in は配備されず、実機動作確認もできない。runtime `Addins\` 反映が必要な場合は `.\build.ps1 -Mode DeployDebugAddIn` を使う
- compile 成功と runtime `Addins\` への実機反映成功を混同しない
- 既存のサービス責務を変更しない
- 新しい処理は既存サービスの責務に合わせて追加する
- UI 制御は専用サービス経由の方針を崩さない
- `WorkbookOpen` に直接依存した表示制御を追加しない
- `ScreenUpdating` を変更した場合は必ず復元する

## 4. 禁止事項

- フローをまたいだ無関係な変更
- UI 表示制御の直接実装
- 既存サービスの責務変更
- `docs/*.md` と矛盾する実装
- 根拠のない推測による修正
- 不明点の自己補完
- 明示指示のない大規模なリファクタリング
- 指示された範囲以外のファイル変更
- 関連が不明確なサービスへの影響拡大

## 5. 不明点の扱い

- 不明な仕様は推測しない
- 不明な事項は必ず「不明」として報告する
- 勝手に補完しない
- 仕様確認が必要な場合は、確認が取れるまで実装方針を確定しない

## 6. 出力ルール

- 変更前に修正方針を説明する
- 変更差分を提示する
- 小さな変更単位で提案する
- 参照した `docs/*.md` を明示する
- docs に記載がない事項は、断定せず「不明」として扱う

## 7. Git運用

- 未コミット変更を混ぜない
- `main` からブランチを切る
- build 成功と実機反映を分けて扱う
